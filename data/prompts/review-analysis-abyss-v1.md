# 네이버 플레이스 리뷰 분석 리포트 작성 지침

> **어비스(ABYSS) 리뷰 분석 파이프라인 표준 프롬프트**
> 버전: v1.0 / 작성일: 2026.05

---

## 🎯 분석 목적

네이버 플레이스 방문자 리뷰 데이터를 기반으로, 매장 사장님에게 실행 가능한 마케팅 인사이트를 제공하는 종합 분석 리포트를 생성합니다.

---

## 📋 입력 데이터 요구사항

### 필수 데이터
- 업체명, 업종, 위치, 메인키워드
- 리뷰 데이터 (최소 50건 이상 권장)
  - 작성자(계정), 방문일, 방문시간대
  - 예약 여부, 대기 시간
  - 방문 목적, 동행
  - 선택 키워드(네이버 플레이스 태그)
  - 방문 횟수, 인증 수단
  - 리뷰 본문

### 선택 데이터 (있으면 더 정확한 분석 가능)
- 답글 여부 및 답글 내용
- 답글 작성일
- 별점 데이터
- 블로그 리뷰 데이터

---

## 🚫 절대 원칙

### 1. 가짜 데이터 금지
- 검색량, 경쟁도, 외부 비교 수치를 임의로 만들어내지 않는다
- 알 수 없는 데이터는 "외부 도구 별도 조회 필요"로 명시
- 추정치·예측치는 "추정"임을 명시 (예: "+X% 예측" 같은 임의 수치 금지)

### 2. 실제 인용만 사용
- 모든 리뷰 인용구는 입력 데이터에서 그대로 복사
- 페르소나/시그널을 위해 인용구를 창작하지 않는다
- 만약 부정 리뷰가 없으면 "부정 사례"를 만들지 말고, 대신 "개선 시그널"로 접근

### 3. 데이터 근거 명시
- 모든 수치는 계산 근거 표시 (예: "5점 비율 76.5% = 78건/102건")
- 모든 인사이트는 출처 데이터 근거 명시

## 📊 리포트 구조 (5개 섹션)

### 섹션 1: 종합 요약 (Executive Summary)

**구성 요소:**
- 매장 기본 정보 (업체명, 업종, 위치, 분석 기간, 분석 리뷰 수)
- KPI 4개 (실제 데이터에서 계산)
  - 5점 비율 (계산: 5점 리뷰 수 / 전체 리뷰 수)
  - 긍정 리뷰 비율 (계산: 4-5점 리뷰 수 / 전체)
  - 핵심 태그 언급률 (가장 많이 언급된 태그의 비율)
  - 재방문률 (계산: 2번째+ 방문 수 / 전체)
- 평점 분포 차트
- 매장 강점 레이더 (6개 영역)
- 월별 리뷰 작성 추이
- 시간대별 방문 패턴 (아침/점심/저녁)
- 방문 목적 분포 (중복 응답 포함)
- 동행 유형 분포
- 방문 횟수 분포 (신규 vs 재방문)

**서술 방식:**
데이터 기반 핵심 진단 (3개 단락)
- 단락 1: 평점·긍정 비율 + 핵심 차별 자산
- 단락 2: 방문 목적·동행 패턴 + 신규/재방문 비율
- 단락 3: 부정 리뷰 분석 + 1점 원인 + 회복 가능성

**잘하는 점 / 개선 필요 점:**
- 각각 4-5개 항목
- 각 항목마다 언급 건수 또는 비율 명시

---

### 섹션 2: 핵심 강점 분석

**4개 카테고리로 분류:**

#### 2-1. 종합 강점 (overall)
- 매장 전체 차원의 핵심 경쟁력
- 강점 2-4개 + 개선점 2-3개

#### 2-2. 매장 환경 (venue)
- 분석 항목: 뷰/위치, 인테리어/분위기, 주차, 청결도, 좌석, 야외 공간 등
- 강점 3-5개 + 개선점 2-4개

#### 2-3. 사장·직원 (staff)
- 분석 항목: 친절도, 응대 속도, 단골 케어, 응대 일관성, 서비스 회복 등
- 강점 2-4개 + 개선점 2-3개

#### 2-4. 메뉴 (menu)
- 분석 항목: 시그니처 메뉴, 베이커리/음식 품질, 가격, 시즌 이벤트, 메뉴 다양성 등
- 강점 3-5개 + 개선점 2-3개

**각 강점/개선점 항목 구조:**
```
- title: 항목명
- score: 점수 (1-100, 언급 빈도 기반 계산)
- mentions: 실제 언급 건수
- desc: 한 줄 요약 (예: "태그 78건 / 본문 22건")
- quotes: 실제 리뷰 인용구 3-4개 (그대로 복사)
- insights: 데이터 분석 인사이트 (왜 중요한지)
- action: 마케팅 활용 또는 개선 액션 (구체적·실행 가능)
```

**점수 계산 기준:**
- 90-100: 매우 강력 (전체 리뷰의 50%+ 언급)
- 80-89: 강함 (30-50% 언급)
- 70-79: 양호 (15-30% 언급)
- 60-69: 보통 (5-15% 언급)
- 60 미만: 약점 또는 개선 필요

---

### 섹션 3: 고객 페르소나 분석

**핵심 페르소나 (3-6개):**

각 페르소나는 다음 기준으로 분류:
- 방문 목적 분포
- 동행 유형
- 방문 횟수 (재방문 패턴)
- 본문에 드러난 정황 (예: "거제 여행", "회사 동료", "엄마와 함께")

**페르소나별 정보 구조:**
```
- name: 페르소나명 (예: "거제도 여행객")
- subtitle: 부제 (예: "관광 일정 중 방문")
- ratio: 비율 추정치 (%)
- icon: 이모지 (시각 식별용)
- visitTime: 방문 시간대
- visitFreq: 방문 빈도
- satisfaction: 만족도 점수 (1-100)
- churnRisk: 이탈 위험 (낮음/중간/높음/매우 높음)
- basis: 분류 근거 (어떤 데이터로 분류했는지 명시)
- motivation: 방문 동기 (3개)
- painPoints: 통증 포인트 (2-3개)
- quote: 긍정 리뷰 인용 (실제 데이터에서)
- improvementSignals: 개선 시그널 배열 (긍정 리뷰 속 숨은 힌트)
  - quote: 실제 리뷰 인용
  - insight: 이 리뷰가 의미하는 개선 시그널
- improvementInsight: 전체 개선 인사이트 요약
- strategy: 마케팅 전략
- retentionAction: 단골 유지 액션
```

**개선 시그널 추출 방법 (핵심):**
- 부정 리뷰가 적은 페르소나도 "긍정 속 숨은 힌트"를 추출
- 예: "비오는 날이라 아쉬웠네요" → 날씨 의존도 시그널
- 예: "정원 있는걸 늦게 알아서 다시 방문했어요" → 안내 부족 시그널
- 만족도 표현 + "~만 빼면", "~했으면 좋겠다", "~가 아쉽다" 같은 미묘한 표현 포착

**이탈 위험 페르소나 (1-3개):**
- 명확한 부정 리뷰가 있는 경우만 별도 분류
- 만족도 50점 이하, 이탈 위험 "매우 높음"
- 회복 가능한 원인 명시 (시스템·정책 문제 등)

---

### 섹션 4: 키워드 분석

> **중요:** 검색량·경쟁도·순위 수치는 절대 임의 생성 금지

#### 4-1. 리뷰 기반 키워드 후보
- 우선순위는 리뷰 언급 빈도 기준
- 각 키워드마다:
  - keyword: 추천 키워드
  - reviewMentions: 실제 리뷰 언급 건수
  - basis: 빈도 근거 (예: '"오션뷰" 22건 + "바다뷰" 18건')
  - strategy: 활용 전략
  - sourceQuotes: 실제 리뷰 표현 인용 2-3개

#### 4-2. 네이버 플레이스 태그 키워드 빈도
- 고객이 선택한 플레이스 태그 전체 리스트
- 빈도 순 정렬
- 워드클라우드 형태 시각화 (빈도에 따라 글자 크기 차등)

#### 4-3. 리뷰 본문 추출 키워드
본문에서 추출한 표현을 그룹별로 정리:
- 뷰 (예: 오션뷰, 바다뷰)
- 분위기 (예: 이국적, 제주도 느낌)
- 포토존 (예: 핑크뮬리, 야자수)
- 메뉴 (예: 시그니처 메뉴명)
- 이벤트 (예: 시즌 이벤트)
- 주변 (예: 인근 명소)
- 부정 (예: 사람 많음, 시끄러움)

**안내 문구 (필수):**
"검색량·경쟁도 외부 데이터는 미포함입니다. 모든 우선순위는 리뷰 본문/태그 언급 빈도 기준입니다. 실제 검색량은 네이버 검색광고 키워드 도구·블랙키위 등으로 별도 조회 후 보완하세요."

---

### 섹션 5: 액션 플랜

**우선순위 3단계:**
- 즉시 (P0): 1주 이내 실행 가능, 효과 즉시
- 1주 (P1): 1주~1개월 내 실행, 효과 중기
- 1개월 (P2): 1개월 이상 소요, 효과 장기

**각 액션 구조:**
```
- priority: 우선순위 (즉시/1주/1개월)
- task: 구체적 액션 (실행 동사로 시작)
- impact: 효과 (낮음/중간/높음)
- effort: 노력 (낮음/중간/높음)
- basis: 데이터 근거 (어떤 리뷰 분석에서 도출되었는지)
```

**액션 도출 원칙:**
- 모든 액션은 분석에서 도출된 데이터 근거가 있어야 함
- 효과/노력 매트릭스로 우선순위 판단
- "효과 높음 + 노력 낮음" 액션 우선

**예상 성과 수치 금지:**
- "+X% 증가", "X위 진입" 같은 예측치는 만들지 않음
- 실제 측정 가능한 지표 (예: "월 리뷰 21건 → 35건")도 추정치임을 명시

## 🎨 시각화 가이드

### 색상 톤
- 긍정/강점: emerald (녹색)
- 개선/주의: amber (황색)
- 부정/위험: red (적색)
- 정보: blue (파랑)
- 차별화/이벤트: purple (보라)

### 차트 종류
- 평점 분포: 가로 막대 그래프
- 강점 레이더: 6개 영역 레이더 차트
- 월별 추이: 영역 차트 또는 막대 그래프
- 시간대별 패턴: 막대 그래프
- 감정 분석: 누적 막대 그래프
- 키워드 빈도: 워드클라우드

### 점수 표시
- 0-100 점수 척도
- 막대 그래프 형태로 시각화
- 그라데이션 색상 (점수에 따라 색상 변화)

## 📦 출력 형식

React 컴포넌트 형태:
- recharts 차트 라이브러리 사용
- lucide-react 아이콘 사용
- Tailwind CSS 스타일링
- 5개 탭으로 구성 (종합요약 / 핵심강점 / 페르소나 / 키워드 / 액션플랜)
- 인쇄용 CSS 포함 (HTML 다운로드 기능)

### 데이터 구조 (TypeScript 타입)

```ts
interface BusinessInfo {
  name: string;
  category: string;
  location: string;
  totalReviews: number;
  avgRating: number;
  period: string;
  mainKeyword: string;
}

interface Persona {
  name: string;
  subtitle: string;
  ratio: number;
  icon: string;
  visitTime: string;
  visitFreq: string;
  satisfaction: number;
  churnRisk: string;
  status: 'core' | 'churn';
  basis: string;
  motivation: string[];
  painPoints: string[];
  quote: string;
  improvementSignals: Array<{
    quote: string;
    insight: string;
  }>;
  improvementInsight: string;
  strategy: string;
  retentionAction: string;
}

interface StrengthCategory {
  label: string;
  icon: ReactNode;
  summary: string;
  items: StrengthItem[];
  improvements: StrengthItem[];
}

interface StrengthItem {
  title: string;
  score: number;
  mentions: number;
  desc: string;
  icon: ReactNode;
  quotes: string[];
  insights: string;
  action: string;
}
```
